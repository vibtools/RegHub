# RegHub v0.3.2.0 Production Readiness

v0.3.2.0 consolidates the v0.3.1.x stabilization line into one clean release. It does not add a
product feature or database migration. It closes production configuration, request-boundary,
local-import, container-package and release-governance gaps discovered during the forensic review.

## Security boundaries

Production startup now requires three independent secrets: `SESSION_SECRET`,
`RUNTIME_ENCRYPTION_KEY`, and `AUDIT_SIGNING_KEY`. Each must contain at least 32 characters and none
may reuse an example value or another boundary's key. Previous key lists and the session-secret
legacy fallback remain available for reading pre-v0.3.2.0 encrypted settings and audit records.

`PUBLIC_BASE_URL`, OIDC endpoints, optional AI endpoints and the screenshot-service endpoint must use
HTTPS in production. Public origin, `ALLOWED_HOSTS`, trusted proxy networks, private-provider
credentials and optional integration dependencies are validated as a complete configuration.

## Input and proxy controls

- Reflected request IDs are bounded to a safe ASCII identifier format.
- Forwarded client chains must contain valid IP addresses and are used only from configured proxies.
- Forwarded host values reject credentials, paths, whitespace and malformed ports.
- Local manifest repository, homepage and screenshot URLs use public-HTTPS SSRF validation.
- Local ZIP metadata rejects path controls, symlinks, encrypted archives, case collisions and limits.

## Container and delivery

The runtime image imports the built `reghub==0.3.2.0` wheel from `site-packages`; it no longer copies
a second `app/` tree over the installed package. CI validates a non-root, read-only container with all
Linux capabilities dropped and `no-new-privileges` enabled. A checked-in SHA-256 manifest locks
all release files and hidden delivery files against accidental post-audit changes.
