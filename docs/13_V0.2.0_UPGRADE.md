# RegHub v0.2.0 Coolify Upgrade

## Zero-freedom compatibility rules

- Preserve the live domain `https://reghub.ygit.dev`.
- Preserve all current Coolify environment values and secrets.
- Preserve Keycloak client ID, callback, role mapping, and session configuration.
- Preserve existing PostgreSQL data and the existing `20260720_0001` migration.
- Do not delete the project directory before replacement; preserve `.git` and local `.env`.
- No deployment logic is added to RegHub. YGIT remains responsible for deployment.

## Before upgrade

1. Create a PostgreSQL backup in Coolify.
2. Record the current successful deployment/commit for rollback.
3. Confirm `/api/v1/health` and `/api/v1/ready` are healthy.
4. Copy the existing source folder to a local backup directory.

## Replace and push

Extract the v0.2.0 ZIP in a temporary folder. Copy the contents of its `reghub` folder over the
existing repository root and choose **Replace**. Do not copy a `.env` file; the release does not
contain one.

```cmd
git status
git add -A
git commit -m "Release RegHub v0.2.0 Smart Registry"
git push origin main
```

## Coolify

Redeploy the RegHub application. The existing entrypoint runs:

```bash
alembic upgrade head
python -m scripts.seed
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Migration `20260720_0002` only adds nullable/defaulted columns and three new tables. It does not
remove or rename existing objects.

```text
https://reghub.ygit.dev/api/v1/health
https://reghub.ygit.dev/api/v1/ready
https://reghub.ygit.dev/api/v1/capabilities
https://reghub.ygit.dev/admin/template/list
https://reghub.ygit.dev/admin/github-import

https://reghub.ygit.dev/admin/local-import
```

Import an Astro repository and a Next.js repository. Confirm Draft status, detected framework and
version, quality score, analysis, Manifest v2, and version history. Select an imported template and
run **Sync Source**; confirm the curated name/status are preserved.

## Rollback

If application verification fails:

1. Use Coolify rollback to the previous healthy image/commit.
2. Do not downgrade the database automatically; v0.1.1 ignores the additive v0.2 columns/tables.
3. Investigate logs before any database downgrade. The v0.2 migration downgrade is provided only
   for controlled maintenance.
