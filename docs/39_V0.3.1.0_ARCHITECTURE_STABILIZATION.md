# RegHub v0.3.1.0 — Architecture Stabilization Release

## Classification

This release is not a feature release. It stabilizes the current RegHub architecture without adding
services, endpoints, integrations, providers, administrator pages, Settings, deployment capabilities
or workflow changes. RegHub remains the YGIT template metadata registry.

## Security stabilization

- Versioned administrator cookies require verified role data; versionless/roleless cookies are rejected.
- `SESSION_SECRET` is deterministically domain-separated for OIDC state, the administrator cookie and
  the SQLAdmin auxiliary session. No new environment value is required.
- Logout clears all RegHub cookies, disables caching and delegates to the existing OIDC end-session
  endpoint when configured.
- GitHub, GitLab and Bitbucket adapters retain private visibility. Private repositories are blocked
  from AI enrichment before the HTTP transport can be created.

## Registry-bound analysis

RegHub continues to detect and retain:

- framework and framework version;
- primary language and package manager;
- license, topics, README-derived title/descriptions and repository metadata;
- preview media, use case, difficulty, quality and security signals.

RegHub no longer generates build, start, runtime, environment, deployment-type or
deployment-readiness recommendations. YGIT owns deployment decisions. Manifest v1/v2 schemas remain
accepted; RegHub-generated manifests are deployment-neutral.

## Database integrity

Migration `20260722_0007_architecture_stabilization` widens one existing identifier column, normalizes
invalid bounded values, removes exact duplicate assets, adds model-aligned constraints, removes
redundant indexes and cleans obsolete generated deployment intelligence. It drops no table or column.

## Repository cleanup

Build/distribution directories, Python/test caches, temporary files and historical generated root
verification/inventory reports are excluded. Historical release manifests and substantive documents
remain available.

## Compatibility lock

All existing public routes, admin pages, Settings, RBAC roles, service tokens, operations, audit
history, provider integrations, Docker, GitHub Actions and Coolify behavior remain in place.
