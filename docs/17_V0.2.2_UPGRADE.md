# RegHub v0.2.2 Coolify Upgrade

## Compatibility rules

- Preserve `https://reghub.ygit.dev`, Keycloak, PostgreSQL, and all current Coolify variables.
- Preserve `.git` and any local `.env` when replacing files.
- Do not delete or recreate the database.
- RegHub remains registry-only; YGIT remains the deployment engine.

## Before deployment

1. Create a PostgreSQL backup in Coolify.
2. Record the current healthy v0.2.1 commit/deployment.
3. Verify `/api/v1/health` and `/api/v1/ready`.
4. Keep `SESSION_SECRET` unchanged. It protects OIDC/admin state and encrypts runtime API secrets.

## Replace and deploy

Extract the replace-ready archive in a temporary directory. Copy the contents of its `reghub`
directory over the existing repository root and choose **Replace**.

```cmd
git status
git add -A
git commit -m "Release RegHub v0.2.2 Operations Console and Runtime Settings"
git push origin main
```

Redeploy in Coolify. The current entrypoint runs:

```bash
alembic upgrade head
python -m scripts.seed
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Migration `20260721_0004` only adds feature, integration, operation, and log tables.

## First verification

```text
/api/v1/health
/api/v1/ready
/api/v1/capabilities
/api/v1/templates?tag=astro
/admin/operations
/admin/settings
```

Then select one template and run **Sync Source**. The browser should open its operation page, show
live progress/logs, and remain there after completion. Use **Back to previous page** to return to the
original template page.

## Runtime credential migration

Existing Coolify secrets continue to work as bootstrap/fallback values. To move a credential to the
Settings page:

1. Open `/admin/settings`.
2. Enter the runtime credential.
3. Optionally disable **Use Coolify ENV as fallback**.
4. Save and apply.
5. Verify the integration status before removing any Coolify value.

No redeploy is needed for Settings changes. Keep at least the initial environment configuration until
the runtime value has been verified.

## Rollback

Rollback to the previous v0.2.1 deployment/commit. Do not automatically downgrade PostgreSQL;
v0.2.1 ignores the additive v0.2.2 tables. Investigate logs before any controlled schema downgrade.
