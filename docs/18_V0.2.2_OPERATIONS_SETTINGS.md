# RegHub v0.2.2 Operations and Settings

## Operations Console

URL: `/admin/operations`

Operations include repository imports, local imports, source synchronization, publication changes,
thumbnail generation, and screenshot retry. Each operation records:

```text
status, progress, requester, input metadata, result, error, return URL,
start/completion timestamps, and ordered log entries
```

The detail page uses Server-Sent Events for one-way live progress/log delivery. It supports copy,
TXT export, cancellation, retry, and a context-preserving return link. Operation data is accessible
only through the authenticated SQLAdmin area.

Queued operations left before execution are recovered after startup. An operation interrupted while
running is marked failed rather than silently re-executed; an administrator can inspect and retry it.

## Feature controls

URL: `/admin/settings`

Every registered feature has:

- **ON/OFF** — whether the feature is available.
- **ALLOW/BLOCK** — whether an administrator may start its task.

Public API controls are runtime switches. Disabling the public API returns a structured HTTP 503 for
registry data endpoints. Health, readiness, authentication, Settings, and existing operation history
remain available so the service can be recovered.

## Integrations

System integrations:

```text
GitHub
GitLab
Bitbucket
AI Metadata
Screenshot Service
```

The Settings page can enable/disable each integration, set its base URL/account/credential, edit
non-secret JSON options, clear the runtime secret, and choose whether Coolify ENV is a fallback.
Custom third-party API configurations can also be added and removed. Custom records provide secure
runtime configuration storage; a project module must explicitly consume a custom integration before
it performs provider-specific work.

The v0.2.2 source adapters continue to target the hosted `github.com`, `gitlab.com`, and
`bitbucket.org` services. Their Settings cards control credentials and state; AI, screenshot, and
custom API cards also accept runtime base URLs.

## Secret handling

- Secrets are encrypted before PostgreSQL storage.
- Secret values are never rendered back into forms or public APIs.
- Leaving a secret field blank preserves its current value.
- **Clear runtime secret** removes only the database value.
- **Use Coolify ENV as fallback** permits the original environment secret when no runtime secret is
  stored.
- Removing a system integration disables it and disables environment fallback.
- Removing a custom integration deletes its runtime record.
- Encryption derives from `SESSION_SECRET`; changing that value prevents decryption of previously
  stored runtime secrets.

## Operational safety

Runtime reload builds new provider clients atomically. Replaced clients remain alive until graceful
application shutdown so an already-running operation is not interrupted by a Settings change.
Provider and custom base URLs must be public HTTPS URLs and cannot contain credentials, custom
ports, blocked local hostnames, or literal private/reserved IP addresses under the existing URL
security policy.
