# RegHub v0.3.0 Safe Upgrade

## Phase A — compatible single-service rollout

1. Back up PostgreSQL and record the healthy v0.2.3.4 commit/deployment.
2. Replace project files while preserving `.git`, local `.env`, Coolify variables and all persistent
   resources.
3. Keep these defaults for the first deployment:

```env
OPERATION_BACKEND=inprocess
CACHE_BACKEND=auto
RATE_LIMIT_BACKEND=auto
```

4. Keep `SESSION_SECRET` unchanged. It remains the fallback decryptor for all existing v0.2.x runtime
   credentials.
5. Commit, push and redeploy the web application.
6. The existing entrypoint runs `alembic upgrade head`, seed and Uvicorn.
7. Verify health, readiness, Keycloak login, Settings, Operations, imports, sync and the public API.

No Redis resource is required in Phase A.

## Phase B — optional Redis worker rollout

After Phase A is healthy:

1. Create a private Redis resource in Coolify with persistence and authentication.
2. Add the same `REDIS_URL` to the web and worker services.
3. Deploy a second application/service from the same commit and image with command:

```bash
python -m scripts.worker
```

4. Set both services to:

```env
OPERATION_BACKEND=redis
CACHE_BACKEND=redis
RATE_LIMIT_BACKEND=redis
```

5. Redeploy the worker, then the web application.
6. `/api/v1/ready` must report a Redis operation backend and an active worker heartbeat.
7. Run one import and one source sync; confirm the worker completes both and live logs remain visible.
8. Change one harmless runtime integration/feature setting and run a new operation; confirm the worker applies the new PostgreSQL-backed setting without a restart.

Do not set `OPERATION_BACKEND=redis` on the web application until a healthy worker is running.

## Key introduction and rotation

Recommended production variables:

```env
RUNTIME_ENCRYPTION_KEY=<independent 32+ character random secret>
RUNTIME_ENCRYPTION_PREVIOUS_KEYS=[]
AUDIT_SIGNING_KEY=<independent 32+ character random secret>
AUDIT_SIGNING_PREVIOUS_KEYS=[]
```

Existing `fernet:v1` values remain readable through the unchanged `SESSION_SECRET`. New or edited
credentials use the primary versioned key.

For a later rotation:

1. Move the old primary value into the appropriate `*_PREVIOUS_KEYS` JSON list.
2. Set the new primary key.
3. Redeploy and verify Settings integrations and `/admin/governance`.
4. Keep previous keys until all records/events signed or encrypted with them have passed the required
   retention period.

## Rollback

Rollback to the v0.2.3.4 commit/deployment. Do not automatically downgrade PostgreSQL; v0.2.3.4
ignores the additive v0.3.0 audit tables, operation role column and indexes. If credentials were edited
under v0.3.0 and therefore use `fernet:v2`, restore the pre-upgrade database backup before relying on
v0.2.3.4 Settings decryption.
