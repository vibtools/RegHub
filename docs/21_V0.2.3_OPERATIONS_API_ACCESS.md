# RegHub v0.2.3 Operations and API Access

## Duplicate imports

Importing a repository that is already registered is a successful no-change outcome:

```text
status: skipped
progress: 100%
outcome: already_exists
```

The existing template is linked in the operation result. Import History records the request as
succeeded with `metadata_snapshot.outcome=already_exists`; it is not recorded as a provider or
application failure.

## Developer operation logs

Repository import and source synchronization logs include ordered stages and diagnostic context:

- command and adapter
- repository URL validation
- import-history identifier
- provider API connection and metadata retrieval
- repository external ID, branch, revision, topics, files, and discovered screenshots
- package manager, language, framework, version, and detection confidence
- category/provider/framework resolution
- manifest/media generation
- database transaction begin/commit
- exception class and exact safe error message

Logs never include provider credentials or service tokens. The operation page keeps SSE as the
primary transport and automatically falls back to incremental JSON polling.

## Operations Console

URL: `/admin/operations`

Available controls:

- search by title, operation type, UUID, or requester
- filter by status and operation type
- newest/oldest ordering
- clear Succeeded, Skipped, Failed, Cancelled, or all terminal history

Queued and Running operations are never removed by Clear History.

## API Manage

URL: `/admin/settings#api-manage-pane`

### Development Mode

Registry endpoints remain publicly readable. Existing feature flags and IP/CIDR/hostname block
rules still apply.

### Live Mode

Registry endpoints require a valid service token. Health and readiness remain public for platform
monitoring.

Supported headers:

```http
Authorization: Bearer vt_reg_xxxxx
```

or:

```http
X-RegHub-Token: vt_reg_xxxxx
```

### Service-token permissions

- `catalog` — templates, manifests, categories, providers, frameworks
- `assets` — template asset endpoints
- `freshness` — template freshness endpoint
- `facets` — catalog facets
- `changes` — incremental change feed
- `capabilities` — capability metadata

A token can be enabled/disabled, deleted, optionally expired, and audited by its last-used time.
The raw token is displayed once. RegHub stores only a keyed digest.

### Block rules

Rules may be an exact IP, CIDR, or hostname. Supported convenience aliases include:

```text
10.x.x.x      -> 10.0.0.0/8
172.x.x.x     -> 172.16.0.0/12
192.168.x.x   -> 192.168.0.0/16
169.254.x.x   -> 169.254.0.0/16
```

`localhost`, `127.0.0.0/8`, `169.254.169.254`, and `metadata.google` are accepted. IP/CIDR rules
match the forwarded client address. Hostname rules match the requested RegHub host. Outbound
integration URLs continue to use the separate SSRF protection boundary.

### API endpoint check

**Check API endpoints** runs authenticated in-process requests against health, readiness,
capabilities, templates, categories, providers, frameworks, and facets. It uses a short-lived
internal check credential and reports HTTP status and duration without exposing a permanent token.
